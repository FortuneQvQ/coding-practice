# 主函数整体逻辑说明

## 一、主函数的职责

`main.cpp` 是整套资讯聚合系统的调度中心。它本身不负责实现所有网络、解析、数据库和页面生成细节，而是把四个模块按正确顺序组织起来：

1. 爬虫模块读取种子列表页、动态接口和新闻详情页。
2. 解析模块从列表页筛选详情链接，从详情页提取标题、时间、正文和图片地址。
3. 数据库模块校验数据、去重入库，并保存每个种子的续爬位置。
4. 展示模块读取数据库，下载或整理图片，生成可搜索、可分类的静态网站。

这种分工使网络失败、页面模板异常或某一条新闻解析失败时，只影响当前请求，不会让整个程序直接退出。

## 二、程序启动阶段

### 1. 确定运行根目录

主函数首先调用 `find_runtime_root`。程序无论从项目目录、Visual Studio 的 `bin/x64/Debug`、`bin/x64/Release`，还是从 `output` 目录启动，都要找到同一份 `config.json`、数据库、模板和输出目录。

确定根目录后，程序把当前工作目录切换到该位置。这样可以避免生成 `output/output`，也可以避免调试版和发布版各自创建一份看似“没有更新”的数据库。

### 2. 解析启动参数

主函数支持以下运行方式：

- 不带参数：抓取资讯、更新数据库，然后生成展示网站。
- `--scan-only`：只验证种子能否发现详情链接并解析出正文，不写入新闻数据库。
- `--generate-only`：不联网爬取，只用现有数据库重新生成展示网站。
- `--no-website`：完成爬取和入库，但不生成静态网站。
- 额外传入一个文件路径：把该文件作为种子配置文件；未指定时使用 `config.json`。

### 3. 初始化公共资源

程序初始化 libcurl 网络环境，读取配置文件中的种子 URL，并打开 SQLite 数据库。数据库打开时会创建缺少的数据表，启用 WAL 模式和忙等待机制，使展示端读取数据库时不必长期阻塞爬虫写入。

随后执行一次历史元数据规范化，把已有记录的来源和主题统一为系统约定的分类。该操作只更新发生变化的记录，不会重复插入新闻。

## 三、核心配额与安全边界

每个种子每次运行最多新增 20 条有效资讯。这里的“新增”必须同时满足：

- 详情页解析成功；
- 标题和正文达到质量要求；
- 文本是有效 UTF-8，不含二进制字节或异常控制符；
- URL 尚未存在于数据库；
- SQLite 确认真正插入了一行。

数据库已有链接、解析失败、乱码、二进制内容和垃圾链接都不会消耗这 20 条新增配额。

为了防止异常网站无限循环，主函数还设置了独立安全上限：单种子最多检查 100 个列表页、最多尝试 80 个详情候选；同一详情 URL 连续失败达到 3 次后会暂时跳过。网络请求之间保留短暂间隔，避免对目标网站造成过大压力。

## 四、每个种子的调度过程

主函数先规范化种子 URL，并用运行期集合删除配置文件中的重复种子。随后由 `crawl_seed` 处理每个唯一种子。

### 1. 读取续爬进度

数据库为每个种子保存两类游标：

- 普通 HTML 列表保存下一页 URL；
- Engine2 动态列表保存下一页页码。

程序仍会检查第一页以发现刚发布的新资讯，但已入库详情会在下载正文前被数据库去重。第一页检查结束后，再从上次保存的位置继续向后翻页，避免每次只重复处理相同内容。

### 2. 抓取种子入口页

`crawl_seed` 调用爬虫模块下载种子 HTML。若入口页抓取失败，本种子记录失败后结束，主循环继续处理其他种子，因此单站异常不会导致整个程序退出。

### 3. 判断列表类型

如果页面脚本包含 Engine2 动态列表参数，主函数进入动态接口流程；否则进入普通 HTML 列表流程。这一判断让 JavaScript 动态渲染页面尽量直接访问其数据接口，而不依赖完整浏览器执行 JavaScript。

## 五、普通 HTML 列表流程

`crawl_html_seed` 负责普通新闻列表、通知中心及其分页页面。

1. 抓取当前列表页。
2. 解析所有锚标签，得到 URL 和锚文本。
3. 使用统一链接规则规范相对路径、限制同域、排除登录页、附件、栏目页、搜索页和分页导航。
4. 对剩余候选评分并排序，优先处理更像真实新闻详情的链接。
5. 调用 `process_candidates` 逐条处理候选。
6. 从当前列表识别真正的“下一页”，而不是首页、尾页或页码导航。
7. 保存下一页 URL，继续翻页，直到新增配额用完、没有下一页或达到安全上限。

如果第一页没有新内容，程序不会把“没有新增”当成成功配额，而会继续检查保存的历史游标和后续页。只有真实新增新闻才会扣减配额。

## 六、Engine2 动态列表流程

`crawl_engine2_seed` 处理部分由 JavaScript 动态加载列表的网站。

1. 从种子 HTML 的脚本配置中提取接口地址、应用编号、引擎实例编号和分类编号。
2. 根据当前页码和每页容量构造表单参数。
3. 由爬虫模块发送 POST 请求，取得 JSON 列表响应。
4. 从 JSON 中提取详情 URL、标题提示和总页数。
5. 把候选交给与普通列表相同的详情处理流程。
6. 成功处理当前页后保存下一页页码，供本次继续翻页和下次运行续爬。

这种方式直接使用页面背后的数据接口，可以解决一部分“查看源代码只有壳，新闻列表要运行 JavaScript 才出现”的页面。若站点没有可识别接口，系统仍会退回普通 HTML 链接筛选逻辑。

## 七、详情候选处理流程

`process_candidates` 是列表发现和新闻入库之间的公共桥梁。普通 HTML 和动态接口得到的候选都会进入这里。

对每个候选依次执行：

1. 规范化 URL，并用本次运行集合去重。
2. 检查数据库是否已有该 URL；已有则记为“已存在”，不重复下载正文。
3. 检查历史连续失败次数，避免对永久无效链接反复浪费请求。
4. 下载详情页，并捕获网络异常。
5. 调用 `parse_detail_page` 解析详情。
6. 调用 `valid_news` 验证标题、正文和 URL 是否符合新闻标准。
7. 由本地分类器规范 `source` 和 `topic`。
8. 调用 `database::addNews` 执行 UTF-8、控制字符、必填字段和唯一 URL 校验。
9. 只有数据库返回“本次真实插入”时，新增计数才加一，并清除该 URL 的旧失败记录。

若下载、解析或校验失败，程序增加失败记录后继续处理下一条候选，不会把异常页面写入数据库，也不会让整个程序退出。

## 八、详情页解析逻辑

`parse_detail_page` 为主函数提供统一解析入口：

- Engine2 或明确包含结构化数据的详情页优先解析 JSON 字段；
- 已知站点使用稳定的标题、正文、时间和图片容器直接提取；
- 其他普通页面使用受异常保护的 HTML 修复和 DOM 解析兜底；
- 如果页面只是列表壳、正文太短或内容无效，则返回失败。

图片型通知不会把图片二进制或 HTML 标签塞进正文。解析器保存图片 URL，展示模块随后把有效图片下载到 `output/images`，数据库继续保存原始图片地址，详情页引用本地相对路径。

## 九、数据库更新与去重

新闻 URL 在数据库中具有唯一约束。程序先查重，再使用 `INSERT OR IGNORE` 作为第二层保护，因此即使同一新闻出现在多个列表页、多个种子或同一次运行的不同分页中，也只会保存一次。

种子游标和详情失败记录也存放在数据库中。正常关闭时，主函数统一调用 `database::closeDb`，不会把数据库永久锁住。WAL 只是一种并发日志机制，不等于锁死数据库。

## 十、展示网站生成

正常抓取结束后，只要没有指定 `--no-website`，主函数调用 `generateNewsWebsite`。展示模块在数据库连接仍可读取时完成以下工作：

1. 读取全部新闻。
2. 创建 `output`、`output/news` 和 `output/images` 等目录。
3. 为每条新闻生成详情 HTML。
4. 下载、校验或复制新闻图片到 `output/images`。
5. 生成首页、搜索页、分类页和前端读取的 `news.json`。
6. 复制项目中已有的正确 JavaScript 和页面模板。

生成结束后主函数关闭数据库并清理 libcurl。展示生成失败会返回独立错误码，方便批处理文件判断原因。

## 十一、扫描验证模式

使用 `--scan-only` 时，程序不会向新闻表插入数据。它会抓取每个种子列表，展示筛选到的详情候选，并抽样下载详情来确认能否解析出真实标题和正文。

该模式适合修改配置或解析规则后先验证站点，不适合代替正式更新。正式入库仍应使用不带 `--scan-only` 的正常运行方式。

## 十二、退出码和最终统计

程序结束前以中文输出：新增入库数量、数据库已存在数量、失败数量、列表页请求数和详情页请求数。

退出码含义如下：

- `0`：本次所选流程成功完成。
- `1`：初始化、配置读取或数据库打开失败。
- `2`：扫描模式中至少一个种子验证失败。
- `3`：爬取和数据库处理已结束，但展示网站生成失败。

项目使用 MSVC `/utf-8` 选项编译，源代码中文字符串按 UTF-8 保存。若旧版 Windows 控制台仍显示乱码，可先执行 `chcp 65001` 再运行程序。

## 十三、整体数据流

```text
config.json 种子
        ↓
main 读取数据库游标并调度种子
        ↓
抓取普通 HTML 列表或 Engine2 JSON 列表
        ↓
筛选、评分并去重详情 URL
        ↓
抓取详情页 → 解析标题/时间/正文/图片
        ↓
质量校验 → 来源与主题归类 → SQLite 唯一入库
        ↓
保存下一页游标和失败状态
        ↓
展示模块读取数据库并生成 output 网站与本地图片
```

## 十四、推荐运行顺序

首次验证配置：

```bat
CampusNewsSystem.exe --scan-only
```

正式抓取、更新数据库并生成网站：

```bat
CampusNewsSystem.exe
```

只根据现有数据库重建网站：

```bat
CampusNewsSystem.exe --generate-only
```

只更新数据库，暂不生成网站：

```bat
CampusNewsSystem.exe --no-website
```
